# Parking Lot

## What this is

A classic facilitation tool, built for the browser. The Parking Lot gives participants a place to drop questions, ideas, and tangents that come up during a session but don't belong in the current conversation. Nothing gets lost, and the discussion stays on track.

The facilitator can return to parked items at any point — usually before the session closes — so participants know their contributions weren't just quietly ignored. It's a small thing that makes a big difference to how safe people feel speaking up.

This tool was built by a lead digital product designer and expert facilitator as part of a growing collection of lightweight workshop tools designed to run in any room, on any device, with no setup.

---

## The tools

**Participant view**
Anyone in the session can open the tool on their own device and submit an item to the parking lot. They enter a short description of what they want to capture, and it appears in the facilitator's view in real time.

**Facilitator view**
The facilitator sees all parked items as they come in, displayed as a running list. Items can be reviewed, addressed, or carried forward at the facilitator's discretion. The view is designed to be readable on a projected screen as well as on a laptop.

---

## How to use

Open `index.html` in a browser — that's it. No installation, no accounts, no configuration.

Participants join on their own devices. The facilitator keeps the tool open on their screen. When an item gets parked, it shows up immediately on the facilitator's view.

At a natural break or before the session closes, the facilitator opens up the parking lot and works through whatever the group surfaced. Some items get addressed on the spot. Others get noted as follow-ups. The point is that every item got acknowledged.

## How to contribute

New tools are welcome, but every addition starts with a completed vibe coding brief before any building begins. The brief captures the facilitation moment the tool is designed for, what the facilitator needs to do or see, what participants experience, and any constraints worth knowing up front. This keeps the toolkit coherent and makes the actual build faster.

If you have an idea for a tool, write the brief first. Then build.

Follow the existing file structure — each tool lives in its own folder with an `index.html` as the entry point. Keep the visual style consistent with the existing tools (see `index.html` in the root of the workshop-tools directory for the design tokens).
